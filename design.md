# 跨平台 YT-DLP 桥接下载器功能规格 V1

## 1. 文档目的

本文档用于替代此前分散的讨论稿、探索稿和旧方案记录，作为后续产品设计、技术选型和实施讨论的唯一功能依据。

本文档只保留已经确认的功能设计、交互语义、边界约束和待后续讨论的技术问题，不再保留旧方案的论证过程。

---

## 2. 产品目标

本产品是一个基于本地桥接服务器和 HTML Web App 的跨平台 yt-dlp 图形界面下载器，目标平台为：

- Windows
- macOS
- Linux

产品目标是同时满足两类使用场景：

- 普通用户可以通过直观界面完成“粘贴链接 -> 选择参数 -> 加入队列 -> 下载 -> 在下载库管理文件”的主流程
- 高级用户可以使用更细粒度的下载选项，包括手动导入 Cookie，以及原始 yt-dlp 参数或格式代码编辑

---

## 3. 总体架构

### 3.1 架构形态

系统采用以下总体结构：

- 本地 Bridge 进程
- 本地 HTTP 接口
- 浏览器中的 HTML Web App
- 本地 yt-dlp 可执行程序
- 本地状态文件与下载库目录

### 3.2 责任边界

Bridge 负责：

- 启动和管理本地 HTTP 服务
- 管理下载任务队列与并发调度
- 调用 yt-dlp 执行解析、下载、续传、删除
- 执行文件系统操作
- 扫描下载库并建立索引
- 维护本地状态持久化
- 实现安全边界与路径校验
- 处理应用级生命周期，例如启动、退出、恢复、后台常驻

Web App 负责：

- 接收用户输入
- 展示元数据、队列、日志、下载库
- 触发任务控制操作
- 提供筛选、搜索、拖拽和确认交互
- 展示错误与状态提示

Web App 不直接负责：

- 调用本地可执行程序
- 直接读写本地任意文件
- 直接操作操作系统资源
- 管理自身静态资源之外的本地持久化

### 3.3 技术选型状态

当前已确认：

- 功能层面采用“跨平台 Bridge + HTML Web App”路线
- Bridge 最终技术栈采用 Rust
- Web App 采用单一 HTML 页面，JavaScript 与 CSS 内嵌，不引入前端框架和运行时依赖
- 本地接口采用 HTTP API
- 本地事件流采用 SSE
- 本地持久化采用拆分 JSON 文件，不使用数据库
- Bridge 按平台分别编译为单一可执行文件
- `yt-dlp` 程序默认与 Bridge 可执行文件放在同目录，便于分发

当前未确认：

- 不在本文档中写死 Rust 侧具体 Web 框架与 HTTP 库
- 不在本文档中展开宿主常驻、托盘和单文件分发的最终实现细节

### 3.4 分发与外部工具约束

V1 的分发策略采用“按平台分发 Bridge 可执行文件 + 同目录 sidecar 工具”的形式。

默认约束如下：

- Windows、macOS、Linux 分别提供对应平台的 Bridge 可执行文件
- `yt-dlp` 默认放置在 Bridge 可执行文件同目录
- 如需完整支持音频提取、合并、转封装等后处理能力，建议将 `ffmpeg` 与 `ffprobe` 也放置在同目录

V1 默认不要求：

- Bridge 自带在线更新 sidecar 工具
- Bridge 自动从网络下载 `yt-dlp`、`ffmpeg`、`ffprobe`

## 4. 产品核心原则

### 4.1 功能优先于技术实现

本文档先锁定产品行为和交互规则，再进入 Bridge 技术选型和实现方案讨论。

### 4.2 队列与下载库分离

“任务队列”和“下载库”是两个不同的系统视图：

- 队列用于展示任务状态、调度状态、恢复状态和下载日志
- 下载库用于展示已经存在于下载根目录内的文件和文件夹

两者可以相互联动，但不混为一个数据模型。

### 4.3 默认安全，允许高级能力

默认界面面向普通用户，主流程尽量直接。

高级功能例如：

- 字幕选项
- 手动导入 Cookie
- 原始 yt-dlp 参数编辑
- 原始格式代码编辑

统一作为高级模式内容，默认折叠隐藏，不干扰普通用户主路径。

### 4.4 明确真实语义，不包装成虚假能力

系统不承诺“真正冻结进程式 Pause”。

运行中的“暂停”在产品语义上定义为：

- Stop：停止当前下载进程
- 保留部分文件
- 允许后续继续下载
- 如续传失败，允许 Restart 或 Delete

因此界面文案采用：

- Stop
- 继续下载
- Restart

而不是：

- Pause
- Resume
- Restart

## 5. 下载根目录模型

### 5.1 全局下载根目录

系统存在一个全局下载根目录。

默认行为：

- 所有下载任务的最终输出位置必须位于该根目录内
- 所有下载库浏览、移动、创建文件夹、删除等操作都限定在该根目录内

### 5.2 子目录规则

用户在创建下载任务时，不是自由输入任意绝对路径，而是：

- 选择根目录本身
- 或选择根目录下已有子目录
- 或在受控规则下创建新的子目录

### 5.3 根目录变更规则

允许用户修改全局下载根目录，但必须满足以下规则：

- 修改前弹出明确确认
- 修改后立即全量重建下载库索引
- 原有下载库视图立即切换到新根目录
- 不自动做旧根目录到新根目录的数据迁移

---

## 6. 用户主流程

用户主流程为：

1. 粘贴单视频或播放列表 URL
2. Bridge 先解析元数据
3. 执行重复检测
4. 确认参数后进入任务队列
5. 调度器按并发限制执行下载
6. 下载过程中可 Stop、继续下载、Restart、Delete
7. 下载完成后文件出现在下载库
8. 用户可在下载库中浏览、打开、移动、删除文件

---

## 7. 页面信息架构

### 7.1 总体布局

页面采用桌面型三段布局：

- 顶部：提交面板
- 左侧主区域：任务队列与任务详情
- 右侧固定栏：下载库

### 7.2 提交面板

提交面板位于页面顶部，直接展示主流程所需核心字段：

- URL 输入框
- 解析按钮
- 下载类型
- 清晰度选择
- 封装格式选择
- 目标子目录选择
- 加入队列按钮

提交面板的主流程交互固定为：

1. 输入 URL
2. 点击解析
3. 查看解析结果与重复检测结果
4. 确认主参数
5. 加入队列

默认显示字段限定为：

- URL
- 解析按钮
- 下载类型
- 清晰度选择
- 封装格式选择
- 目标子目录选择
- 加入队列按钮
- 更多选项入口

其中：

- 并发数属于全局设置，不作为提交面板中的单任务字段
- “先解析、再入队”是固定交互，不采用“直接提交后后台再解析”的主界面模式

解析完成后，提交面板下方应展示解析确认条，至少包括：

- 标题
- 缩略图
- 提取器类型
- 时长
- 预估大小
- 是否为播放列表
- 播放列表条目数
- 重复检测结果

当命中重复检测时，解析确认条应直接提供：

- `Skip`
- `Add Anyway`
- `Open Existing`

高级项收纳到“更多选项”：

- 字幕下载
- 手动导入 Cookie
- 原始 yt-dlp 参数编辑
- 原始格式代码编辑
- 其他未来扩展项

高级项展开后的布局固定分为以下 3 个分组：

- 字幕与语言
- Cookie 与认证
- 原始参数覆盖

高级项区域应同时展示明确风险提示，说明原始参数和格式代码可能影响默认去重、续传和后处理行为。

### 7.3 主工作区

左侧主工作区分为上下两层：

- 上层：任务队列
- 下层：当前选中任务的详情与日志

详情区采用双页签结构：

- `详情`
- `日志`

V1 不采用多面板堆叠详情模式，避免同屏信息过载。

### 7.4 下载库栏

右侧固定栏展示下载根目录下的树状下载库。

其定位是：

- 文件浏览
- 目标目录选择数据源
- 受限文件管理

下载库栏固定分为三段：

- 顶部工具条
- 中部树状视图
- 底部当前选择区

顶部工具条至少包括：

- 刷新
- 新建文件夹
- 设为目标目录

底部当前选择区用于承接当前选中文件或文件夹的上下文操作。

## 8. 任务模型

### 8.1 任务类型

系统至少支持两类任务：

- 单视频任务
- 播放列表父任务

播放列表父任务下可包含多个子视频任务。

### 8.2 任务状态机

V1 采用以下状态集：

- `submitted`
- `resolving`
- `queued`
- `downloading`
- `stopping`
- `paused`
- `completed`
- `failed`
- `canceled`

状态语义：

- `submitted`：用户已提交，等待解析
- `resolving`：Bridge 正在解析元数据
- `queued`：已入队，等待并发槽位
- `downloading`：yt-dlp 正在下载
- `stopping`：用户请求停止，系统正在终止进程并落盘状态
- `paused`：进程已停止，部分文件被保留，可尝试继续下载
- `completed`：下载完成
- `failed`：下载失败
- `canceled`：用户在未完成前取消

### 8.3 任务元数据

解析阶段至少应拿到以下元数据：

- 标题
- 提取器类型
- 视频唯一 ID
- 时长
- 上传者或来源信息
- 缩略图
- 预估大小
- 可选格式信息
- 输出路径
- 是否为播放列表
- 播放列表名和子项数量

解析完成后才允许进入正式队列。

---

## 9. 队列与并发模型

### 9.1 队列原则

队列默认采用 FIFO 语义。

调度器按以下条件拉起任务：

- 任务状态为 `queued`
- 当前活动下载数小于全局并发上限
- 任务未被停止、删除或取消

### 9.2 并发数

系统支持用户设置最大同时下载数。

默认建议值为：

- `3`

### 9.3 并发变更原则

当用户修改并发数时：

- 新限制对后续调度立即生效
- 已经启动的任务不强制中断
- 超过新上限的情况不通过强制停止来回收并发

---

## 10. Stop、继续下载、Restart、Delete 语义

### 10.1 Stop

Stop 的定义是：

- 停止当前 yt-dlp 子进程
- 保留 `.part`、片段文件和必要状态
- 将任务转入 `paused`

Stop 不是操作系统级进程挂起，不承诺网络连接被冻结后原地恢复。

### 10.2 继续下载

“继续下载”表示：

- 重新拉起下载进程
- 使用续传参数尝试基于已有部分文件继续下载
- 续传是否成功取决于源站、协议、时间跨度和认证状态

系统应明确告诉用户：

- 继续下载通常可行，但不是 100% 保证
- 如果续传失败，可改用 Restart

### 10.3 Restart

Restart 的定义是：

- 删除该任务相关的部分文件
- 保留任务元数据
- 以全新下载方式重新开始

### 10.4 Delete

Delete 的定义区分两类场景：

- 队列删除：从队列和状态中移除任务
- 文件删除：删除磁盘中的对应部分文件或成品文件

对于未完成任务，Delete 默认行为是：

- 删除任务记录
- 删除部分文件
- 不保留该任务的未完成中间结果

对于已完成任务，Delete 文件表示：

- 删除磁盘成品文件
- 更新下载库索引
- 从相关展示中移除该文件

---

## 11. 程序重启与任务恢复

### 11.1 恢复策略

程序重启后，未完成任务采用全局设置控制。

默认策略：

- 默认不自动继续下载
- 恢复为“可手动继续”的状态

### 11.2 恢复结果

重启后系统应能识别并恢复：

- 原 `queued` 任务
- 原 `paused` 任务
- 原因异常退出导致中断的下载任务

对于异常退出前处于下载中的任务：

- 系统应将其恢复为可继续处理的未完成状态
- 不默认假设它们已经完成

---

## 12. 播放列表规则

### 12.1 播放列表显示方式

播放列表在队列中显示为父任务。

父任务可展开后查看子任务列表。

父任务至少展示：

- 播放列表名
- 总条目数
- 已完成数
- 当前执行子项
- 整体进度摘要

### 12.2 展开方式

用户点击父任务后可展开或收起其子任务。

子任务仍然是独立下载单元，但属于该父任务上下文。

实现约束：

- 播放列表父任务是聚合视图，不直接占用下载并发槽位
- 真正占用并发槽位的是其下的可执行子任务

### 12.3 下载落点规则

播放列表下载时，系统规则为：

- 用户先选择一个目标子目录
- 在该子目录下自动创建一个“播放列表名文件夹”
- 该播放列表的各个子任务文件默认输出到这个文件夹内

### 12.4 大播放列表确认

对于较大的播放列表，系统应在入队前做明确确认，避免用户误操作触发超大量下载。

---

## 13. 重复检测

### 13.1 主判定键

V1 重复检测的主判定键为：

- `extractor + videoId`

### 13.2 数据来源

重复检测结合以下来源：

- 队列内现有任务
- 下载库索引
- 根目录内扫描得到的 sidecar 信息，例如 `.info.json`

### 13.3 命中后的默认动作

检测到同一视频已经存在于下载根目录任意子目录时，默认动作为：

- `Skip`

并向用户提供两个备选操作：

- `Add Anyway`
- `Open Existing`

### 13.4 重复但仍允许加入的场景

以下场景允许用户显式添加重复任务：

- 目标子目录不同
- 输出格式不同
- 清晰度不同
- 用户明确希望保留多个版本

### 13.5 不采用的策略

V1 不将“文件重命名后仍能完美识别重复”作为保证目标。

因此：

- 下载库不提供重命名能力
- 重复检测主要依赖提取器和视频 ID，而不是纯文件名猜测

---

## 14. 下载库设计

### 14.1 角色定位

下载库是下载根目录的受控树状视图，职责是：

- 浏览目录结构
- 选择目标目录
- 打开文件或文件夹
- 在根目录内部进行受限移动
- 删除文件或文件夹
- 创建文件夹

### 14.2 不与任务状态混合

下载库不直接等同于任务历史。

一个文件可以：

- 存在于下载库中
- 但不再显示为活跃任务

一个任务可以：

- 存在于队列历史中
- 但对应文件已被删除

### 14.3 树结构规则

下载库采用树状结构，默认行为为：

- 文件夹优先
- 文件在后
- 隐藏部分文件、临时文件和内部状态目录
- 支持展开和收起文件夹
- 支持单选和多选

### 14.4 自动刷新规则

下载库刷新规则如下：

- 由下载任务引发的变化自动刷新
- 外部程序直接改动文件系统时，不做强实时监听保证
- 外部变更通过手动刷新更新到界面

### 14.5 打开行为

双击文件时：

- 视频或音频文件使用系统默认程序打开

双击文件夹时：

- 展开或收起该文件夹

### 14.6 创建文件夹

用户可在下载库中创建新文件夹，但必须保证：

- 新文件夹位于下载根目录内
- 名称合法
- 不与现有路径冲突

### 14.7 删除规则

下载库允许删除文件或文件夹，但必须满足：

- 删除前弹出明确确认
- 删除后立即更新索引和界面
- 删除行为仅限于下载根目录内

### 14.8 重命名规则

V1 不支持下载库内重命名。

原因：

- 会破坏重复检测的稳定参照
- 会引入额外的路径冲突和同步复杂度
- 不是 V1 的核心能力

---

## 15. 下载库中的拖拽移动

### 15.1 支持的能力

下载库支持：

- 单选拖拽移动
- 多选拖拽移动

### 15.2 限制条件

仅允许以下移动：

- 从下载根目录内移动到下载根目录内的其他目录
- 移动已完成文件
- 移动普通文件夹

不允许以下移动：

- 将文件移出下载根目录
- 将文件移入自身子目录
- 移动进行中的下载文件
- 移动到已存在同名目标的位置并自动覆盖
- 通过拖拽触发自动重命名

### 15.3 冲突处理

当目标位置已存在同名文件或文件夹时：

- 直接提示冲突
- 拒绝本次移动
- 不自动重命名
- 不默认覆盖

---

## 16. Web App 队列交互

### 16.1 队列区

队列区至少包含：

- 队列标题
- 任务总数
- 状态筛选
- 搜索
- 批量操作入口
- 任务列表主体

队列列表的视觉密度采用紧凑列表，不采用大卡片式任务块。

单个任务项默认使用双行信息结构：

- 第一行：状态、标题、类型、目标目录摘要、关键操作
- 第二行：进度、速度、ETA、排队信息或错误摘要

批量操作在 V1 先限定为：

- 批量删除
- 批量继续下载
- 批量 Restart

V1 不要求批量 Stop。

### 16.2 任务行内容

任务行应根据状态展示不同信息。

下载中任务至少显示：

- 标题
- 类型
- 当前进度
- 下载速度
- ETA
- 目标目录
- 关键操作按钮

下载中任务默认关键操作为：

- `Stop`

排队任务至少显示：

- 排队状态
- 排队顺序
- 目标目录
- 关键参数摘要

排队任务默认关键操作为：

- `Delete`

暂停任务至少显示：

- 已有进度
- 继续下载按钮
- Restart 按钮
- Delete 按钮

失败任务至少显示：

- 错误摘要
- Retry 或继续下载
- Restart
- Delete

完成任务至少显示：

- 完成状态
- Open File
- Open Folder
- Delete

播放列表父任务使用聚合行展示，不采用与单视频任务完全相同的行结构。

播放列表父任务至少显示：

- 播放列表名
- 总条目数
- 已完成数
- 当前执行子项
- 整体状态摘要
- 展开或收起入口

播放列表父任务默认不承载子任务级别的 Stop、继续下载、Restart 操作；这些操作由子任务行承载。

### 16.3 详情与日志区

选中任务后，下方详情区至少展示：

`详情` 页签至少包括：

- 原始 URL
- 保存位置
- 元数据摘要
- 最近错误
- 提取器类型与视频 ID
- 上传者或来源信息
- 输出参数摘要
- 当前或最后一次状态摘要

`日志` 页签至少包括：

- 下载日志
- 最近错误相关日志片段
- 最近一次下载尝试的关键事件

日志区默认采用滚动文本视图，V1 不要求富文本日志分析能力。

## 19. 启动与运行体验

### 19.1 首次启动

首次启动时：

- 启动本地 Bridge
- 自动打开浏览器
- 引导用户确认或修改全局下载根目录
- 由 Bridge 直接提供内嵌的单页 Web App

### 19.2 后续启动

后续是否自动打开浏览器由用户设置控制。

### 19.3 持续运行

浏览器关闭不应中断下载任务。

Web App 与 Bridge 是分离的：

- 关闭浏览器标签页不等于停止 Bridge
- 重新打开 Web App 后应看到当前最新任务状态

前后端通信规则：

- 普通操作通过 HTTP API 提交
- 队列状态、下载进度、日志增量和下载库刷新事件通过 SSE 推送

## 20. 本地持久化

V1 使用本地持久化保存以下信息：

- 全局设置
- 队列任务
- 任务状态
- 恢复信息
- 输出路径信息
- 播放列表父子关系
- 下载库索引

持久化要求：

- 写入尽量原子化
- 程序异常退出后可恢复
- 不依赖外部数据库服务

### 20.1 状态目录规则

运行时状态文件默认存放于当前用户的数据目录，而不是 Bridge 可执行文件所在目录。

这样做的原因包括：

- 避免程序目录写权限问题
- 避免覆盖程序文件时误删状态文件
- 使设置、任务状态和下载库索引与分发文件解耦

### 20.2 JSON 拆分原则

V1 的本地持久化采用拆分 JSON 文件结构，不使用数据库。

建议的持久化拆分方向如下：

- `settings.json`：全局设置
- `queue.json`：队列顺序与轻量状态摘要
- `tasks/<taskId>.json`：单任务完整信息
- `playlists/<playlistId>.json`：播放列表父任务聚合信息
- `attempts/<taskId>/<attempt>.json`：单次下载尝试记录、错误摘要与日志摘要
- `library/index.json`：下载库索引缓存
- `runtime/session.json`：上次运行会话恢复信息

### 20.3 写入与恢复原则

JSON 文件的写入应尽量遵守以下规则：

- 优先更新最小必要文件，而不是重写全部状态
- 通过临时文件写入后原子替换，降低损坏风险
- `library/index.json` 视为可重建缓存，不作为唯一真实来源
- 任务恢复以任务文件和运行会话文件为主

---

## 21. 安全边界

系统必须遵守以下安全边界：

- 仅允许受控 URL 类型
- 文件系统操作不得越出下载根目录
- 创建目录时过滤非法名称
- 对 Cookie 文件导入做路径和格式校验
- 日志中避免暴露敏感内容
- 并发数应有上限
- 不允许通过前端直接执行任意本地命令
- sidecar 工具调用路径应受控，默认仅使用可执行文件同目录或用户显式配置的工具路径

## 26. 总结

本产品 V1 的最终方向已经明确：

- 产品形态是跨平台 Bridge + HTML Web App
- Bridge 技术栈采用 Rust，并通过本地 HTTP API + SSE 对浏览器提供能力
- Web App 采用单一无依赖 HTML 页面，JavaScript 与 CSS 内嵌
- 本地持久化采用拆分 JSON 文件，不依赖数据库
- Bridge 按平台生成可执行文件，并默认与同目录 `yt-dlp` 协同工作
- 产品体验是普通用户可直接使用，同时保留高级能力
- 核心能力是“先解析、再入队、可恢复、可去重、带下载库”
- 下载控制语义采用 Stop、继续下载、Restart
- 下载库允许删除，不允许重命名

后续所有讨论默认以本文档为准。
